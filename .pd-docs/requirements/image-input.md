# PRD：聊天输入框支持图片输入

> 状态：Draft（待与设计/体验/实现汇总） · 维护：王产品 · 2026-06-30
> 关联路线图：不在 `docs/roadmap.md` 现有 Phase 1–3 范围内，属**新提议**。建议作为 Phase 2「丰富体验」的一个子项推进（见 §优先级）。

---

## 1. 背景与问题

club 当前是一条**纯文本管道**。代码核查结论：

- 契约（`packages/shared/src/types.ts`）：`Message.content: string`，`CreateMessageRequest = z.object({ content: z.string().min(1).max(4000) })`——消息体只有一根字符串。
- 后端（`packages/server/src/routes/messages.ts` + `db.ts`）：`messages.content TEXT`，无上传/附件相关路由，无 `multipart` 处理。
- 前端（`packages/web/src/components/composer.tsx`）：单一 `<Textarea>`，`onSend(content: string)`；`renderContent`（`lib/format.tsx`）只解析 `@mention` 高亮，没有任何富文本/附件/图片节点。
- 三端对等性现状：CLI（`club send`）、MCP（`send` tool）、Web 全部只接受/产出字符串 content。

**问题**：在「人与 agent 平等参与」的房间里，真实协作中大量信息是**视觉性**的——截图报 bug、贴设计稿、发一张白板照片、贴一张错误截图让 agent 诊断。这些今天只能靠「描述图片」，或把图片丢到外部链接再贴 URL。前者丢失信息，后者把信息搬出 club，破坏「同一套消息历史」的承诺。

**结合灵魂判断**：图片输入强化（而非稀释）人机平等——它让一条消息能承载**人与 agent 都能消费**的信息载体。真正的张力在「agent 能不能『看见』」，这是本 PRD 要拍板的核心决策（见 §4）。

## 2. 目标与非目标

### 目标
1. 房间内的参与者（人 / agent）能**发送图片**，图片作为消息的一部分进入统一历史。
2. 所有参与者（人 / agent）能在**自己的入口**看到/获取这张图片：Web 渲染、CLI/MCP/SDK 能取到图片的引用。
3. 三端契约**对等**：图片是消息的一个字段，不是「Web 专属特性」。

### 非目标（明确不做，防蔓延）
- **不做** 视频、音频、文件（任意附件）传输。本 PRD 只覆盖图片。
- **不做** 图片编辑、裁剪、标注、滤镜、贴纸。
- **不做** 内联 OCR、内联图片搜索、图片内的 @mention 解析。
- **不做** 历史消息的富文本重渲染（旧的纯文本消息不变）。
- **不做** 端到端加密 / 私密图片（与 club 当前的明文存储模型一致；见 §7）。

## 3. 用户与场景

| 场景 | 谁发 | 给谁看 | 价值 | 是否本期 |
|---|---|---|---|---|
| 人贴 bug 截图，@agent 让它诊断 | 人 | agent + 人 | 把「描述」变「看见」，诊断更准 | ✅ 核心 |
| 人贴设计稿/白板照讨论 | 人 | 人（agent 旁听） | 视觉协作 | ✅ |
| agent 贴它生成的图表/截图作为回复 | agent | 人 | agent 也能用视觉输出，不只是文字 | ⚠️ 见 §4 决策（MVP 仅「能发」，不强求生成） |
| agent 真正「看懂」图片内容并推理 | — | — | 多模态 agent 协作 | ❌ 见 §4 决策（明确排除出 MVP） |

## 4. 关键决策：图片要不要让 agent「真正看见」？

> 这是范围与复杂度的分水岭。给明确判断。

**判断：MVP 只做「图片作为可共享、可展示的消息载体」，不做「图片作为 agent 的多模态推理输入」。**

理由：
1. **价值优先级**。今天 club 最痛的不是「agent 看不懂图」，而是「根本发不出图」。先把「人能贴、所有人能看」这一层做出来，已经解锁了上面场景表里前两行（占真实使用的大头）。
2. **复杂度量级完全不同**。
   - 仅展示：后端加一个图片存储 + 一个 URL/ID 字段；前端加一个 `<img>` 渲染；CLI/SDK/MCP 在 content 旁附带一个结构化引用。范围可控。
   - 真正「看见」：要让 agent 入口（CLI/MCP）把图片**下载下来、转成 base64/多模态 block、喂进底层 LLM**。这要求 (a) SDK/CLI 知道当前 agent 用的是哪个支持 vision 的模型，(b) 一套把图片从 club 历史搬运进 agent 上下文的机制，(c) 成本/尺寸/速率的全新治理面。这是一个**独立且庞大的特性**，不应混进「图片输入」MVP。
3. **不破坏灵魂**。把图片做成「same backend, same history」里的一等消息字段，对人和 agent **对称**——人能渲染、agent 的客户端也能取到引用。agent「暂时不解析」是能力问题，不是权限问题。等真要让它「看见」时，契约不用动，只动 agent 客户端的取图逻辑。**这把决策做成了可演进、不返工的一步。**
4. **决策可逆、可验证**。先落地展示层，观察真实使用里「agent 被要求看图」的频率，再决定是否启动「多模态输入」专项。比一次梭哈更稳。

**落地含义（约束实现）**：
- 图片在契约里是消息上的一个**结构化引用**（如 `attachments: [{ type: "image", url, width?, height?, mime?, bytes? }]`），不是塞进 `content` 的 base64。
- `content` 仍是必填的人类/agent 可读文本（可简短，如「看看这张截图」），保证 CLI 纯文本回看、MCP tool 入参、agent 不解析图片时**仍有可理解的内容**。图片是 content 的**补充**，不是替代。
- agent 客户端本期**不要求**下载/理解图片，但契约里必须**保留**足够信息（url + 尺寸），使未来多模态可直接复用。

## 5. 功能需求（按端）

### 5.1 后端（`packages/server` + `packages/shared`）
- **F-B1 契约扩展**：`Message` 增加可选 `attachments?: Attachment[]`；`Attachment` 至少含 `type: "image"`、`url`（可解析为完整 URL）、`mime`、`bytes`、可选 `width/height`。
- **F-B2 上传**：`POST /uploads`（鉴权）接受图片，返回 `{ attachment }`（含 url）。限定允许的 mime（jpeg/png/gif/webp）与单图尺寸上限。
- **F-B3 发消息**：`CreateMessageRequest` 扩展为接收 `attachments`（引用已上传图片的 id/url），与 `content` 一起落库；`content` 仍 `min(1)`。
- **F-B4 读取**：`GET /messages` / SSE 返回的消息包含 attachments；历史一致。
- **F-B5 存储位置**：图片落本地磁盘目录（可配置路径，docker volume 挂载），通过 server 的静态路由提供访问；不引入对象存储依赖（与 club 当前 SQLite 单文件的轻量部署一致）。

### 5.2 Web（`packages/web`）
- **F-W1 输入**：Composer 支持选择/粘贴/拖拽图片；发送前先 `POST /uploads`，再带 attachment 发消息。
- **F-W2 预览**：选中后、发送前在输入框内显示缩略图预览，可移除。
- **F-W3 渲染**：消息列表中带 image attachment 的消息，在文本气泡内/下方渲染 `<img>`（懒加载、点击放大、带 alt/尺寸约束、不撑破布局）。
- **F-W4 失败兜底**：上传失败/图片加载失败时给出明确错误反馈，草稿（含已选图）不丢失。

### 5.3 CLI / SDK / MCP（对等性）
- **F-C1 SDK/契约消费**：SDK 的 `Message` 类型带 attachments；纯文本格式化（`format.ts`）在 content 后附 `[图片: <url>]`，保证 CLI 回看、`club listen`、MCP 拉历史时**看得见图片的存在与链接**。
- **F-C2 CLI 发送（最小）**：`club send` 至少支持通过选项附带一张本地图片（如 `--image path`），底层走 upload + send。MVP 可不做 TUI 内粘贴，但命令行附图必须可用，否则三端不对等。
- **F-C3 MCP tool**：`send` tool 的入参 schema 增加 optional `images`（路径或 url），使 agent 能发出图片引用；返回与 Web 一致。
- **F-C4 agent 理解图片**：**本期明确不做**（见 §4）。MCP/CLI 仅产出/展示引用，不下载喂模型。

## 6. 非功能需求

- **尺寸/配额**：单图上限建议 ≤ 5MB；单条消息图片数 ≤ 4（防滥用与渲染压力）；mime 白名单。
- **性能**：图片经静态路由直出，不走 SSE 走 SQL；消息列表懒加载缩略图，首屏不被大图阻塞。
- **安全/隐私**：与 club 当前模型一致——存储明文、key 即权限。上传需鉴权；静态访问是否要求 key 由后端定（开放问题 §8）。**不引入**加密。
- **可靠**：上传与发消息解耦——上传失败不发消息；发消息失败保留草稿（含图）。
- **可观测**：上传计数、失败率进入现有日志通道。

## 7. 验收标准（可测）

> 这些条目直接对应测试用例与人工验收，王测开据此写测试、王产品据此验收。

- **AC-1（契约）**：`packages/shared` 的 `Message`/`CreateMessageRequest` 含可选 attachments；含 attachments 的消息能被 zod 校验通过；不含 attachments 的旧消息结构仍 100% 兼容（现有测试全绿）。
- **AC-2（后端上传）**：鉴权后 `POST /uploads` 对 jpeg/png/gif/webp ≤ 5MB 返回 201 + attachment（含可访问 url）；超大或非法 mime 返回 4xx；未鉴权返回 401。
- **AC-3（发消息）**：带 attachments 的 `POST /messages` 落库并在 `GET /messages` 与 SSE 中原样返回；`content` 为空时仍被拒绝（content min(1) 不变）。
- **AC-4（Web 发送）**：选择/粘贴/拖拽一张图 → 输入框出现可移除缩略图 → 发送 → 历史中出现该消息且图片可见。上传失败时草稿不丢、有错误提示。
- **AC-5（Web 渲染）**：带图消息在窄/宽屏下不溢出、加载失败有占位、点击可放大、尊重 reduced-motion（如有放大动效）。
- **AC-6（三端对等）**：
  - `club send --image <path> "说明"` 发送后，Web 历史能看到该图。
  - `club listen` / SDK 纯文本回看里，该消息文本含 `[图片: <url>]` 标记（不会「看不见」）。
  - MCP `send` 带 `images` 参数能成功发送且 Web 可见。
- **AC-7（边界/滥用）**：单条消息附 > 4 张图被拒；超 5MB 被拒；非图片 mime 被拒；错误均为可理解的 4xx + 明确文案。
- **AC-8（回归）**：现有纯文本消息发送、@mention、SSE、CLI/MCP 全链路功能不被破坏（现有测试套全绿）。

## 8. 优先级与依赖

- **P0**：F-B1/F-B2/F-B3/F-B4 + F-W1/F-W3 + AC-1~AC-5、AC-7、AC-8。即「Web 能发能看、后端契约闭环、不破坏现状」。这是不可再分的最小有价值切片。
- **P1**：F-C1（SDK/CLI 看得见引用）、F-C2（CLI 附图）、F-C3（MCP 附图）、AC-6。即三端对等。**强烈建议与 P0 同期或紧随**，否则会出现「Web 能发图、CLI/MCP 看不见」的断层，违背 same-history 承诺。
- **P2**：F-W2（预览/移除）、F-W4 细节、缩略图/点击放大打磨（这部分体验细节交王体验、视觉交王设计）。
- **不做（后续专项）**：agent 多模态「真正看图」（独立 PRD）、视频/音频/文件、加密图片。

依赖：无硬前置；建议与 Phase 2「丰富体验」并行启动。后端契约改动（F-B1）是所有端的阻塞前置。

## 9. 风险与开放问题

- **存储与成本**：本地磁盘 + volume 在单机部署够用；多实例/水平扩展时需迁对象存储——MVP 不做，但 F-B5 的存储层应抽象到一处，便于未来替换。（架构层面归王架构。）
- **静态访问鉴权**：图片 URL 是否需要 key 才能访问？开放 → 简单但泄露风险；鉴权 → CLI/MCP 取图要带凭证。**建议 MVP：开放静态访问但 URL 含不可猜测的随机 id**（security through obscurity 的轻量方案），正式鉴权列为开放问题。（安全层面需与王后端确认。）
- **滥用/spam**：一个大图能撑爆 SSE 历史与移动流量。靠尺寸/数量上限 + 未来可加速率限制。
- **隐私**：截图常含敏感信息，而 club 当前明文存储且无删除能力。**开放问题**：是否需要「撤回/删除图片消息」？建议作为紧随其后的 P1（涉及整条「消息删除」能力，超出本 PRD）。
- **agent 多模态演进**：若 §4 排除的「真看图」需求来得很快，契约（attachments 结构）需在那时证明已足够（已含 url/尺寸/mime），无需破坏性改动——这是本设计预留的演进口。
- **i18n**：上传/失败的文案需进 i18n 字典（zh/en）。
