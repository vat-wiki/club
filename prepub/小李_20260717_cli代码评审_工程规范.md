# 代码质量与工程规范全面评审 — club-cli

> **整理时间：** 2026-07-17
> **评审范围：** `/packages/cli/src/` 下全部 30 个源文件（17 个命令 + 5 个核心模块 + 4 个测试 + 1 入口 + 1 TUI）
> **一句话摘要：** 架构设计出色（依赖注入、fail-open 更新机制），但存在**错误处理路径不一致**、**`any` 类型滥用**、**重复格式逻辑**等工程债务，核心命令缺测试覆盖。

---

## 一、代码重复

### 1.1 错误处理模式的重复（6 处）

每个命令 action 内部几乎都有完全相同的 `catch → console.error → process.exit(1)` 三段式：

| 文件 | 行号 | 模式 |
|------|------|------|
| `commands/whoami.ts` | 13–16 | `console.error((err as Error).message)` + `process.exit(1)` |
| `commands/info.ts` | 39–42 | 同上 |
| `commands/members.ts` | 16–19 | 同上 |
| `commands/cat.ts` | 59–62 | 同上 |
| `commands/update.ts` | 26–29 | `console.error(\`error: ${...}\`)` + `process.exit(1)` |
| `commands/join.ts` | 104–107 | 同上（但 `update.ts` 前缀了 `"error: "`，其余没加） |

**问题：** `index.ts:24–29` 已定义了顶层 `die()` 函数统一格式化错误为 `error: <msg>`，但 6 个命令绕过了它，自行 `catch` 后格式也不统一（有的加 `error:` 前缀，有的不加）。

**建议：** 命令 action **不要自行 catch**，直接 `throw`，让 `program.parseAsync().catch(err => die(err))`（`index.ts:87`）统一处理。这会让 `whoami/info/members/cat/update/join` 与 `rooms/enter`（已让异常冒泡）保持一致。

### 1.2 尾部斜杠剥离逻辑（4 处）

`.replace(/\/$/, "")` 出现在 4 个文件中：

- `commands/login.ts:10`
- `commands/join.ts:54`（`runJoin` 内部）和 `88`（action 内部——**同一文件内重复了两次**！）
- `commands/recover.ts:21`

**问题：** `join.ts:54` 在 `runJoin` 内部 strip 一次，但 `join.ts:88` action 传给 `runJoin` 前又 strip 一次——**双重 strip**，虽然语义正确，但逻辑重复且容易出 bug。

**建议：** 提取纯函数 `stripTrailingSlash(url: string): string`，在 `config.ts` 或新文件 `url.ts` 中定义。`join.ts` 的 action 层应去掉 strip，只由 `runJoin` 负责。

### 1.3 `formatMessageWithIds` 重复实现（1 处）

`commands/format.ts:1–3` 仅做 re-export：
```ts
export { formatMessage } from "@club/sdk";
```

但 `commands/read.ts:7–32` 定义了一个 `formatMessageWithIds`，**手抄了 `formatMessage` 的全部格式化逻辑**（时间格式、图标、deleted 标记、media 拼接、reactions），只是额外加入了 attachment id 显示。

**问题：** 
- 两处逻辑不同步：SDK 侧的 `formatMessage` 更新后，CLI 的副本不会同步。
- 函数参数类型为 `any`（见下方类型安全章节）。

**建议：** 在 `format.ts` 中实现 `formatMessageWithIds` 时，**复用** SDK 的 `formatMessage`，仅追加 id 显示部分；或向上游 SDK 提 PR，让 `formatMessage` 支持 `--show-ids` 选项。

## 二、错误处理

### 2.1 错误处理策略不一致

| 策略 | 命令 |
|------|------|
| **自行 catch + console.error + exit(1)** | whoami, info, members, cat, update, join |
| **直接 throw，依赖顶层 die()** | rooms, enter, delete, react, search, whoami（矛盾） |
| **完全无 try-catch** | login（action 同步，理论上不会抛）, listen, mentions |

**数据：** 17 个命令中，6 个自行 catch、6 个让异常冒泡、5 个无任何防护。三条路径并存是工程隐患。

**建议：** 统一为"抛异常 → 顶层 `die()` 处理"模式。`die()` 已经正确格式化并 exit(1)，无需每处重复。

### 2.2 `listen.ts:25` — 静默吞掉 `reportAgentThinking` 错误

```ts
// src/commands/listen.ts:23-26
const reportThinking = (m: Message) => {
  if (!mention || !mentionMatches(m.content, mention)) return;
  void client.reportAgentThinking(m.room).catch(() => {});  // ← 空 catch
};
```

**问题：** `.catch(() => {})` 完全吞掉错误，没有任何日志。虽然报告思考状态是 best-effort，但**生产排查时毫无线索**。

**建议：** 至少 `.catch((e) => console.error('club: reportThinking failed:', e.message))`。

### 2.3 `saveConfig` 无错误处理

`src/config.ts:58–62`：
```ts
export function saveConfig(cfg: ClubConfig): void {
  const p = configPath();
  if (!existsSync(dirname(p))) mkdirSync(dirname(p), { recursive: true });
  writeFileSync(p, JSON.stringify(cfg, null, 2) + "\n", "utf8");  // ← 可能抛
}
```

`login/recover/join/enter` 的 action 全部调用 `saveConfig`，但**没有任何一处 try-catch**。磁盘满、权限不足、目录不可写时，会抛未捕获异常 → `process.exit(1)` 但错误信息可能不友好。

**建议：** `saveConfig` 应 catch 并抛 `ConfigError`，或 action 层显式保护。

### 2.4 `mentions.ts:35–38` — error 类型推断不严谨

```ts
// src/commands/mentions.ts:35-38
} catch (err) {
  const status = (err as { status?: number }).status;
  if (status !== 409) throw err;
}
```

**问题：** 用 `as { status?: number }` 断言，但 `err` 可能是 `ClubApiError`（已有 `status` 属性）。此处应直接 `err instanceof ClubApiError` 判断。

## 三、类型安全

### 3.1 `read.ts:7` — 最严重的 `any` 滥用

```ts
// src/commands/read.ts:7
function formatMessageWithIds(m: any): string {  // ← any 参数
```

**问题：** 函数参数是 `any`，意味着类型系统完全放弃对该函数的保护。调用方传入错误结构时无任何编译期提示。

**影响范围：** `read.ts:48` 调用处传入的是 `ClubClient.messages()` 的返回结果（类型为 `Message`），但函数签名说"什么都接受"。

**建议：** 改为 `(m: Message): string`，直接从 `@club/shared` 导入 `Message` 类型。

### 3.2 `read.ts:18` — `any` 在 map 回调中重复出现

```ts
// src/commands/read.ts:18-23
const media = (m.attachments ?? [])
  .map((a: any) => {  // ← any
    if (a.mime.startsWith("video/")) ...
  })
```

**问题：** 如果 `m` 改为 `Message` 类型，这里的 `a` 自然就是 `Attachment`，无需 `any`。

### 3.3 `stdin.ts:56` — 强制类型断言

```ts
// src/commands/send.ts:56
content = await readStream(process.stdin as unknown as ReadableLike);
```

**问题：** 两次类型断言。`process.stdin` 确实是 Node Readable，`ReadableLike` 接口也窄到足以匹配。虽然运行时没问题，但两次 `as unknown as` 是类型断言的反模式标志。

**建议：** 让 `readStream` 接收 `NodeJS.ReadableStream`（或直接 `Readable`），减少断言。

### 3.4 `update.ts:80–81` — JSON.parse 后强制断言

```ts
// src/update.ts:79-82
const obj = JSON.parse(raw);
return obj && typeof obj === "object" ? (obj as UpdateCache) : {};
```

**问题：** 即使 `typeof obj === "object"`，也可能是数组或其他对象字面量，`as UpdateCache` 没有验证字段类型。不过因为 `UpdateCache` 所有字段都是 optional，运行时不会崩，属于"弱安全"。

### 3.5 `update.ts:110` — fetch 返回类型断言

```ts
// src/update.ts:109-111
const json = (await res.json()) as { version?: unknown };
const v = json?.version;
```

**问题：** fetch API 的 `res.json()` 返回 `Promise<unknown>`，此处断言合理。但后续 `typeof v === "string"` 的检查是运行时兜底，说明**类型层面没有保障**——可考虑 zod schema 校验。

### 3.6 `send.ts:70` — send deps 传参类型不匹配

```ts
// src/commands/send.ts:70
send: (c, ids, r) => client.send(c, ids, r ? { room: r } : undefined),
```

`SendDeps.send` 签名（`send-impl.ts:20`）：
```ts
send: (content: string, attachmentIds?: string[], room?: string) => Promise<unknown>;
```

此处当 `room` 为 `undefined` 时传入 `undefined`，符合签名。**无问题**，但 `undefined` 作为 room 参数的含义依赖于服务端默认行为——类型上无法约束。

## 四、配置管理

### 4.1 `config.ts` 整体评价 — 良好，2 处可改进

**做得好的：**
- Zod schema 校验 + `safeParse` → 类型安全强
- `CLUB_CONFIG` 环境变量覆盖 → 测试友好
- `requireConfig()` 抛 `ConfigError` 而非返回 null → 语义清晰
- `defaultRoom()` 纯函数 → 可测试

**改进建议：**

**4.1.1 `loadConfig` 未处理 `readFileSync` 异常**

```ts
// src/config.ts:52-56
export function loadConfig(): ClubConfig | null {
  const p = configPath();
  if (!existsSync(p)) return null;
  return parseConfig(readFileSync(p, "utf8"));  // ← 权限/编码错误会抛
}
```

`existsSync` 和 `readFileSync` 之间有 TOCTOU 竞态（文件可能在二者之间被删除）。但因为是 CLI 单次运行，实际风险低。建议加 try-catch 返回 null。

**4.1.2 配置项是否完整？**

当前 `ClubConfig` 只有 `server`、`key`、`room` 三项。`limit.ts` 的默认限制（50/500）是硬编码常量，不从配置读取。如果未来需要支持用户自定义 `maxLimit`、`defaultRoom` 等，应提前预留扩展接口。

### 5.3 `search.ts:20` 与 `parseLimit` 行为不一致（实测数据）

```ts
// search.ts:20 — 手写的 limit 解析
const limit = Math.min(Math.max(1, Number(opts.limit) || 20), 100);

// limit.ts — 共享的 parseLimit
// Number("") || 50 → 50
// Number("0") || 50 → 50（因为 0 是 falsy！）
```

**实测差异表：**

| 输入 | `search.ts` 结果 | `parseLimit()` 结果 | 是否一致 |
|------|-----------------|---------------------|---------|
| `"0"` | 20（`0 \|\| 20`） | 1（clamp up） | ❌ **不一致** |
| `""` | 20（`NaN \|\| 20`） | 50 | ❌ **不一致** |
| `"abc"` | 20（`NaN \|\| 20`） | 50 | ❌ **不一致** |
| `"-5"` | 1 | 1 | ✅ |
| `"500"` | 100（clamp） | 500 | ❌ **上限不一致** |

**结论：** `search.ts` 应该用 `parseLimit()`，而不是手写。当前两者默认值（20 vs 50）和上限（100 vs 500）都不一致，用户体验割裂。

## 六、依赖注入 / 解耦

### 6.1 当前架构评分：★★★★☆（4/5）

**优秀的模式：**
- `join.ts`：`JoinDeps` 接口 + `runJoin(input, deps)`，action 只负责 wiring ✅

**仍需改进的（直接在 action 中 new ClubClient）：**

| 命令 | 耦合方式 | 影响 |

| `whoami.ts` | `new ClubClient(cfg).me()` | 无法 mock，需真实网络 |
| `info.ts` | `new ClubClient(cfg)` | 同上 |
| `rooms.ts` | `new ClubClient(cfg).rooms()` | 同上 |
| `members.ts` | `new ClubClient(cfg).members()` | 同上 |
| `react.ts` | `new ClubClient(cfg).toggleReaction(...)` | 同上 |
| `delete.ts` | `new ClubClient(cfg).deleteMessage(...)` | 同上 |
| `search.ts` | `new ClubClient(cfg).search(...)` | 同上 |
| `cat.ts` | `new ClubClient(cfg)` | 同上 |
| `listen.ts` | `new ClubClient(cfg).stream(...)` | 同上 |
| `update.ts` | 无（已解耦 ✅） | — |
| `recover.ts` | `new ClubClient({ server })` | 同上 |
| `login.ts` | 无（不调用 SDK） | — |

**数据：** 14 个有 API 调用的命令中，**3 个已解耦**（join/enter/send），**11 个未解耦**。

### 6.2 `tui.tsx` 的 4 个 ClubClient 实例

```ts
// src/tui.tsx:28, 43, 58, 129
new ClubClient(cfg)  // ← 出现 4 次
```

**问题：** `useEffect` 初始化（L28）、加载历史消息（L43）、订阅流（L58）、发送消息（L129）分别创建 4 个独立的 `ClubClient` 实例。虽然 `clientRef` 保存了 L28 的实例，但 L43/L58 创建了新实例，L129 又创建了一个。

**建议：** 在 `useEffect` 中创建一次 `ClubClient`，通过 `useRef` 或 `useState` 共享给所有逻辑，避免 4 个独立的 WebSocket 连接。

## 七、其他发现

### 7.1 `tsconfig.json` — 测试文件被排除

```json
// tsconfig.json:10
"exclude": ["src/**/*.test.ts", "dist"]
```

**问题：** `.test.ts` 文件不参与类型检查（`typecheck` 脚本）。如果测试代码有类型错误，`npm run typecheck` 不会报错。当前测试都用了正确的类型（得益于 `vitest` 自身的检查），但这是隐式保障。

**建议：** 要么把测试纳入 typecheck，要么在 CI 中显式跑 `tsc --noEmit` 包含测试。

### 7.2 `listen.ts:41–44` — SIGINT handler 泄漏风险

```ts
// src/commands/listen.ts:41-44
process.on("SIGINT", () => {
  sub.stop();
  process.exit(0);
});
```

每次调用 `makeListenCommand` 都会注册一个 SIGINT handler。虽然 commander 通常只创建一次命令实例，但如果将来有测试场景或重复调用，handler 会堆积。**建议：** 用 `once` 或在 cleanup 中移除。

### 7.3 `send.ts:53` — stdin 自动检测的边界

```ts
// src/commands/send.ts:53
const useStdin = opts.stdin || (!text.length && !process.stdin.isTTY);
```

当 `text=[]`（无参数）、`isTTY=true`（交互式终端）时，`useStdin=false`，内容将为空字符串 → `runSend` 抛出 "no message"。**行为正确**，但错误信息可以更具引导性（提示用户用 `--stdin` 或传入文本）。
