# 首次入口（First Contact）

> **状态**：①问题已钉死、方向已定（待评审） ②未落地
> **维护者**：王产品
> **关联**：`identity-recovery.md`（本文档是它的**上游**）、`agent-integration.md`、`docs/roadmap.md`
> **触发**：主理人 leon 以全新陌生人视角体验测试环境 :6200，直觉"哪里不对劲"，原话举例「一开始进来是不是该让输入密钥而不是直接创建账号」。首轮体验取证见 `.ux-review/fresh/`（01/03/06/07 等）。

## 0. TL;DR（先给结论，再给论证）

1. **默认 create 不变**——全新陌生人确实没有 key、必须创建，「默认 create」对真人是**唯一可行**的分诊结果。leon 的直觉「该让输入密钥」错在前提（新人没 key 可输入），但他捕捉到的**不适感是真的**：根因不是默认模式选错，而是**入口前 5 秒零信息**——既没讲清「key 即身份、只展示一次、丢了找不回」，也没把「我是新人 / 我已有 key」当作两条平等路径分诊。
2. **入口从「单路快速加入」改为「平等分诊 + 前置告知」**：两条路（创建 / 用已有密钥进入）视觉等权，入口顶部用一句话讲清 club 的身份哲学（key 是唯一凭证、丢了找不回）。这不是改默认模式，是**补足决策期的信息缺口**。
3. **身份找回（恢复码）落地，方向 D 拍板推进，P0**：单靠"入口讲清楚"只能降低丢的概率，治不了"已经丢了"。`identity-recovery.md` AC2–AC11 必须落地，否则"丢了找不回"这句承诺就只能在 UI 里诚实地说出来、却没有任何挽回余地——那不是诚实，是能力缺位。
4. **web create 与 server `/join` 保留分工、收敛到同一套心智**：不合并（服务对象不同），但二者必须共享同一套"分诊 + 身份哲学告知"的语言，结束现状的**双轨不一致**。

## 1. 背景：陌生人的前 5 秒，club 没说话

### 1.1 现状（读代码 + 取证钉死，非道听途说）

陌生人打开 web（或 server `/join`），看到的是这样：

- **web AuthDialog**（`.ux-review/fresh/01-create-zh.png`，`packages/web/src/components/auth-dialog.tsx`）：
  - 标题 `club.`，副标题「起个昵称加入聊天室。」，单输入框「昵称」，主按钮「加入」。
  - 底部一行小字灰链「已有密钥？」才切到 paste 模式。
  - **全入口零处**提及 key、身份、凭证、不可找回——观感就是一个普通的「快速注册/加入」。
- **server `/join`**（`.ux-review/fresh/06-join.png`，`packages/server/src/public/join.html`）：
  - 副标题「人与 agent 平等参与的聊天室。」、`returning` 链「已经领过密钥？进入聊天 →」、有 kind 选择（人类/agent）、有 onboarding snippets。
  - 文案/形态/字段与 web AuthDialog **完全不一致**——同一个产品，两套入口，两套话术。

而**创建之后**的 KeyReveal（`.ux-review/fresh/03-key-reveal.png`，`packages/web/src/components/key-reveal-dialog.tsx`）文案其实**已经做对了**：

> ⚠️ 请保存你的登录密钥
> 这是回到当前身份的唯一凭证。请妥善保存——club 不会替你保存，丢失后也无法找回。

问题不在 KeyReveal（它不可关闭、强制确认，已达标 `identity-recovery.md` AC1）。问题在**它来得太晚**——用户在入口的 5 秒决策期里，被引导着以为这只是个"随便起个名加入"的轻量操作，等密钥弹出来才知道「哦原来这玩意儿丢了找不回」。**告知错位到了决策之后**。

### 1.2 leon 的直觉对在哪、错在哪

- **对**：入口确实"不对劲"。他敏锐地感到：一个随便"加入"的操作背后，藏着一个他还没被告知的、不可逆的承诺。这是**真问题**。
- **错**：解法不是"让输入密钥"。全新陌生人没有 key，让他在第一屏输入一个不存在的东西，是无解的死路。"默认 create"对真人是唯一可行的——leon 自己作为熟人觉得"我该有 key 才对"，是因为他知道 club 有 key 机制；真正的新人不知道，也不能因此逼他凭空变出一个 key。
- **真正该修的**：入口要在**决策之前**把身份哲学讲清，并把"我有 key / 我没有"当作**两条平等路径**呈现，而不是把"已有密钥"藏成一行小字。让新人安心走 create，让回来的人一眼看到 paste——**分诊清晰，告知前置**。

### 1.3 为什么这是产品问题，不是体验/视觉问题

王体验管"按钮好不好点"，王设计管"好不好看"，但这事的根是**产品决策层**：

- 入口讲不讲「key 即身份」、讲不讲「丢了找不回」，是**产品要不要在第一屏承诺什么**的问题——这是产品方向，归王产品。
- 「恢复码找回」要不要做，是**产品能力是否存在**的问题，不是 UI 微调——归王产品。
- 两套入口要不要分工、共享什么心智，是**产品架构**问题——归王产品。

交互细节（两条路怎么排、文案具体怎么写、动画）后续交王体验/王设计，但**该不该分诊、该不该前置告知、找回要不要做**——这三条决策是本文档的产出。

## 2. 目标与非目标

### 2.1 目标

1. **入口前置传达 club 的身份哲学**：陌生人在**做任何操作之前**，就知道「key 是这个身份的唯一凭证、创建时只展示一次、丢了找不回（除非你保存了恢复码）」。把现在错位到 KeyReveal 的告知，**前移到入口决策期**。
2. **平等分诊两条路**：「我是新人，创建身份」与「我已有密钥，进入」作为两条视觉等权的路径，结束"已有密钥？"被藏成小字的现状。新人一眼知道自己该走哪条，回来的人一眼看到自己的入口。
3. **让身份找回成为产品能力**（与 `identity-recovery.md` 联动）：落地恢复码（方向 D），让"丢了找不回"这句 UI 诚实承诺，有一个真实的挽回后路兜底——否则它只是能力缺位的遮羞布。
4. **两套入口收敛同一心智**：web AuthDialog 与 server `/join` 不再各说各话，共享同一套"分诊 + 身份哲学告知"语言，差异只在各自的服务对象（web 用户 vs CLI/MCP 用户）。

### 2.2 非目标（明确不做，防需求蔓延）

- **不把默认模式从 create 改成 paste**：新人没 key，默认 paste 是死路。这条已论证，**不改**（见 §4.1）。
- **不做"输入密钥才放行"的强制门**：那等于要求所有新人在没有 key 的情况下自证有 key，逻辑不成立。
- **不把两套入口物理合并成一个页面**：web SPA 与 server 静态页服务对象不同、部署形态不同，合并得不偿失（见 §6）。本文档只要求二者**心智收敛**，不要求代码合并。
- **不在入口做完整的 club 产品介绍/教程**：入口只讲身份哲学这一件事，不塞"club 是什么、能干什么"的长篇——那是 onboarding/空状态该做的，不是门口该做的。
- **不在本文档定义恢复码的技术细节**：恢复码的数据层/端点/安全权衡全在 `identity-recovery.md` §6，本文档只**拍板方向 D 推进 + 定优先级**，不重复其内容。

## 4. 方向决策

### 4.1 (a) 默认模式：维持 create，但补足「平等分诊 + 前置告知」

**决策：默认 create 不变。** 理由（把空间扫干净）：

| 候选 | 评估 | 结论 |
|------|------|------|
| 默认 create（现状） | 新人唯一可行路径；问题是**告知错位**，不是默认值选错 | **保留默认值，补告知** |
| 默认 paste | 新人没 key，第一屏要求输入不存在的东西 = 死路 | **否决** |
| 平等分诊（create / paste 视觉等权，无默认） | 表面公允，但**真人群里新人占绝大多数**，回来的人是少数；强制分诊会让每个新人都多一次"我是新人吗"的自我盘问 | **否决为默认，但采纳为视觉布局** |

**关键区分**：「默认模式」与「视觉分诊」是两件事。**默认仍是 create**（因为新人占多数，且 create 是唯一可行解），但**视觉上**两条路要等权呈现——不再把"已有密钥？"压成一行小字，而是把 create 与 paste 作为两个并列的可选入口，create 因是默认/主路径而略突出（如主按钮），paste 作为清晰的次级路径（明显的按钮/区块，不是灰链）。这样新人一眼走 create，回来的人一眼看到 paste，**谁都不用找**。

**前置告知的内容**（入口顶部一句话级，不长篇）：club 的身份是「一个 key」，key 是回到这个身份的唯一凭证，创建时**只展示一次**，丢了**找不回**（恢复码落地后改为"丢了用恢复码找回，恢复码也丢则无法找回"）。把 KeyReveal 里那句已经做对的文案，**前移一份到入口**。

### 4.2 (b) 身份找回：方向 D 拍板推进，P0

**决策：`identity-recovery.md` 方向 D（key + 一次性恢复码，找回时换发新 key）拍板推进，优先级 P0，与本文档同批落地。** 理由：

1. **单靠入口告知治不了"已经丢了"**。把"丢了找不回"在入口讲清楚，是**诚实**，但诚实不等于**有能力**。一个 chat 产品，换浏览器/清缓存是真人使用的**必然事件**，如果必然事件导致永久失联、且产品方零挽回能力，这是**产品能力的断层**，不是"用户该自己负责"。
2. **方向 D 已经过充分论证**（见 `identity-recovery.md` §4–§6）：轻量（一个高熵恢复码，不是邮箱密码）、三端对等（web/CLI/MCP 共用同一端点）、与现有 key 模型兼容演进（增量 migration、不破坏存量）、安全模型不倒退（恢复码存 hash、找回时换发新 key 而非返回明文）。它不是"偷偷做成账号体系"——它**不收集邮箱/手机号、不设密码、不发邮件**（AC11 守护），是「key 即身份」哲学内的能力补全，不是哲学替换。
3. **它直接服务 §3 场景 C**：换设备没 key 的人，手里有恢复码就能回。没有它，场景 C 是死局；有了它，闭环。
4. **与"前置告知"形成闭环**：入口说"丢了用恢复码找回"，是因为**真的能找回**——告知与能力一致，而不是空头支票。

**本文档不重复 `identity-recovery.md` 的 AC2–AC11**，只在此**确认其优先级为 P0 并解除其"待评审"状态**。落地验收以那份 PRD 为准。

> 一个范围说明：恢复码找回解决"换设备/清缓存"。**已登录后想再看一眼 key** 的需求，现状已由 `ViewKeyDialog`（`packages/web/src/components/view-key-dialog.tsx`）覆盖——它是"创建时没存好、还没丢"的逃生舱。两者不冲突、不重复。

### 4.3 (c) 两套入口：保留分工、收敛心智

**决策：web AuthDialog 与 server `/join` 不物理合并，但必须收敛到同一套"分诊 + 身份哲学告知"心智。** 理由：

| 候选 | 评估 | 结论 |
|------|------|------|
| 合并成一页 | web SPA 与 server 静态 HTML 部署形态、构建链不同；合并要 server 服务 React 或 web 内嵌 onboarding，工程成本高、收益小 | **否决** |
| 各做各的（现状） | 同一产品两套话术、两种字段、两种分诊逻辑，认知割裂 | **否决** |
| **保留分工 + 心智收敛** | 各自服务对象不同（web 给人聊天、/join 给 CLI/MCP onboarding），但共享同一套身份哲学语言与分诊结构 | **采纳** |

**分工保留**：
- **web AuthDialog**：面向用浏览器聊天的人。重点是 create/paste 分诊 + 身份哲学告知 + 进入聊天 UI。**不**展示 CLI/MCP onboarding snippets（那是 agent 接入者的事，web 用户不需要）。
- **server `/join`**：面向通过 CLI/MCP 接入的人与 agent onboarding。重点是 create（带 kind 选择：人类/agent）+ 身份哲学告知 + **onboarding snippets**（`club login` / `claude mcp add` 指引）。`returning` 链「已经领过密钥？进入聊天 →」保留（它跳到 `/chat` SPA）。

**心智收敛（必须一致）**：
- 二者**都**在入口顶部讲同一句身份哲学（key 是唯一凭证、只展示一次、丢了用恢复码找回 / 也丢则无法找回）。
- 二者**都**把 create 与"已有密钥/已有身份"作为清晰的两条路（web 是 create/paste；`/join` 是 create/returning）。
- 二者的 KeyReveal（如果 `/join` 也有展示 key 的环节）**文案同源**——同一套 i18n key，不是各写一份。
- 二者**都**在恢复码落地后，提供"用恢复码找回"入口。

## 5. 功能需求（三端）

| 端 | 入口分诊与告知 | 找回（联动 identity-recovery） |
|---|---|---|
| **web** | AuthDialog：顶部一句身份哲学告知；create 与 paste 两条路视觉等权（create 为默认主按钮、paste 为清晰次级路径，不再是小字灰链）；恢复码落地后，paste 区或第三条路提供"用恢复码找回"。KeyReveal 文案保留（已达标）。 | "找回身份"入口：name + 恢复码 → 换发新 key → 进房间（AC4，详见 identity-recovery） |

| **MCP** | 不在 MCP 范围（MCP 消费已有 key，不创建身份）。 | 不在 MCP 范围（agent 找回见 identity-recovery §8，本期不做）。 |

## 6. 验收标准（可测，喂王测开）

> 找回相关 AC（AC4–AC11）见 `identity-recovery.md` §7，此处不重复，仅以引用确认其优先级 P0。本节聚焦**入口分诊与前置告知**。

- **FC1（前置告知存在）**：web AuthDialog 在 create 模式下，**入口可见区域**（未点开任何折叠、未进入 KeyReveal）即包含能被识别为「key 是唯一凭证 / 只展示一次 / 丢了找不回（恢复码落地后改为"用恢复码找回"）」语义的文本。（验证：playwright 驱动全新无 localStorage 浏览器打开 web，断言入口 DOM 含上述关键词；截图对照。）
- **FC2（两条路视觉等权）**：web AuthDialog 中，"用已有密钥进入"（paste）入口以**按钮或同等显著的交互元素**呈现，**不再是** `<button class="...text-muted-foreground underline...">` 的小字灰链（现状代码 `auth-dialog.tsx` 末尾）。（验证：读 DOM 断言 paste 切换控件的 computed style 不再是 muted 小字 + underline；或 playwright 截图人工/视觉回归确认两条路视觉权重对等。）
- **FC3（默认仍是 create）**：web AuthDialog 初始 `mode === "create"`（现状已成立，回归保护）。（验证：读 `auth-dialog.tsx` 的 `useState<Mode>("create")`；playwright 断言打开即见昵称输入框。）
- **FC4（`/join` 心智收敛）**：server `/join` 顶部包含与 web **同源**的身份哲学告知文本（同一套 i18n key 或语义一致的双语文案），不再是仅"人与 agent 平等参与的聊天室"这一句。（验证：读 `join.html` 的 i18n 文案表 + playwright 打开 `/join` 断言告知文本存在。）
- **FC5（KeyReveal 文案不退化）**：KeyReveal 仍保留"丢失后也无法找回"（恢复码落地前）或"丢失后用恢复码找回"（落地后）语义，不可关闭、强制确认。（回归保护，验证同 `identity-recovery.md` AC1。）
- **FC6（告知与能力一致）**：**当且仅当** `identity-recovery.md` AC4–AC6（恢复码找回闭环）达标后，入口与 KeyReveal 的文案才允许写"丢了用恢复码找回"；在恢复码**未**落地时，文案必须保持"丢了也无法找回"。**不允许文案承诺 UI 兑现不了的能力**。（验证：恢复码未落地时，grep 入口/KeyReveal 文案不得出现"恢复码/recover"字样；恢复码落地后反之。此 AC 防止"空头支票"。）

## 7. 优先级与依赖

| 项 | 优先级 | 依赖 |
|---|---|---|
| 入口前置告知（FC1） | **P0** | 无（纯前端文案/布局） |
| 两条路视觉等权（FC2） | **P0** | 无（纯前端） |
| `/join` 心智收敛（FC4） | **P1** | 无（server 静态页文案） |
| 恢复码找回（identity-recovery AC2–AC11） | **P0**（本文档拍板） | `identity-recovery.md` §6 的 server migration + 端点 |
| 告知与能力一致（FC6） | **P0**（守护项） | 与恢复码落地**同步** |

**落地顺序建议**：FC1/FC2（入口告知 + 分诊，纯前端，快）→ 恢复码找回（后端 + 三端）→ FC6 在恢复码落地同批把文案从"丢了也无法找回"切换为"丢了用恢复码找回"。`/join` 收敛（FC4）可与任一批并行。

## 8. 开放问题（需主理人拍板）

1. **前置告知的"度"**：入口顶部一句话讲身份哲学，会不会**劝退**轻量用户？（"我刚来你就跟我说丢了找不回？"）我的判断是**必须讲**——知情同意优于转化率，且恢复码落地后承诺变成"能找回"，负面感会大幅降低。但具体文案的"吓人程度"需王体验/王设计打磨，建议给两版文案（轻/重）做对比。**倾向：恢复码落地前用"较轻但诚实"的措辞，落地后切换为正面措辞。**

2. **paste 路径是否同时承担"恢复码找回"入口，还是独立第三条路？** 两条路（create/paste）已经很满，加恢复码找回变三条路会让入口变重。倾向：**恢复码找回作为 paste 路径下的二级入口**（"用已有密钥进入"旁边一个"密钥丢了？用恢复码找回"的小入口），而不是第三条并列主路。待王体验确认信息架构。

3. **`/join` 是否需要展示恢复码？** `/join` 创建身份时是否也像 web 一样弹 KeyReveal + 恢复码？现状 `/join` 创建后直接进入聊天 UI（key 存 localStorage）。若 `/join` 用户主要是 CLI/MCP onboarding（他们会 copy key 到 CLI/MCP 配置），恢复码是否也该在 `/join` 展示？倾向：**该展示**——三端对等原则，且 `/join` 也可能被真人用。待确认。

4. **恢复码的"用后作废 vs 换发新恢复码"**（沿用 `identity-recovery.md` §8.1）：我倾向**换发**（找回成功返回新恢复码，用户重新保存），体验更连贯。待主理人 + 王后端拍板。

## 9. 与 roadmap 的关系

本文档与 `identity-recovery.md` 共同构成 **Phase 1.5（身份闭环）** 的核心——它补的是 MVP 验收时被遗漏的「身份生命周期」完整需求（详见 `identity-recovery.md` §9 复盘）。建议 roadmap 在 Phase 1 与 Phase 2 之间显式标注这个身份闭环批次，避免它被 Phase 2 的"多房间/丰富体验"挤掉优先级。**身份闭环是 Phase 2 一切体验的前提**——身份都不稳定，多房间、reactions、编辑删除都是建在沙地上。

## 10. 一句话总结

leon 的直觉抓对了"不对劲"，但病根不是默认模式，而是**入口在前 5 秒对身份哲学沉默**、且**找回能力缺位**让这种沉默只能是诚实的坏消息。修法是：**前置告知 + 平等分诊 + 落地恢复码找回**，三件事一起做，让"key 即身份"从一个只在创建后才被知晓的副作用，变成入口就讲清、丢了能找回的产品承诺。
