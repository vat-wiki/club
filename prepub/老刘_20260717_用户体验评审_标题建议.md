> **来源**：/home/leon/projects/club/packages/cli/src/ 全量源代码（29 个文件）

## 一、命令提示和错误信息是否友好？

### ✅ 做得好的

- **`index.ts:26-29`** 全局 `die()` 统一错误格式 `error: <msg>`，理念正确。
- **`send-impl.ts:54-56`** 空消息时的错误提示极其友好，直接把用户可能的三种解法都列出来了：
  > `"no message. pass text, use --stdin, or attach --image/--video/--file <path>"`
- **`stdin.ts:24-27`** `--stdin` 用了但没管道输入时，不仅报错，还给了完整示例：
  > `"--stdin requires piped input (e.g. echo hi | club send --stdin); stdin is an interactive terminal"`
- **`join.ts:46-51`** 名字冲突时用自定义 Error 包装，输出 `"name \"X\" already taken; choose another"`，比原样抛 409 体面多了。
- **`enter.ts:44-48`** 房间名字非法时客户端直接拦截，提示 `"must be 1-30 chars of [a-z0-9-], starting alphanumeric"`，不用等服务器 400。

### ⚠️ 需要改进的

| # | 问题 | 位置 | 用户视角 |
|---|------|------|----------|
| 1 | **错误格式不统一** | 多个命令文件 vs `index.ts:die()` | 全局 `die()` 加 `error:` 前缀，但 `whoami.ts:14-15`、`join.ts:104-106`、`cat.ts:59-62`、`members.ts:17-18` 等都用 `console.error(err.message)` 直接输出，没加 `error:` 前缀。用户在不同命令间切换时，报错风格不一致，显得零散。 |
| 2 | **`join` 命令错误丢失上下文** | `join.ts:104-106` | 直接 `console.error(err.message)`，如果 SDK 抛的是 `"Fetch failed"` 这种底层信息，用户完全不知道是网络问题还是名字问题。对比 `index.ts:die()` 会带 `error:` 前缀 + 进程退出码 1，`join` 的 action 自己 `process.exit(1)` 但前面没格式化。 |
| 3 | **鉴权过期时的提示太干** | `config.ts:72-75` / `index.ts:83` | 全局兜底是 `"not logged in. run: club login <key>"`。但如果 key 真的过期（401），用户看到的是 SDK 原始错误（如 `"401"` 或 `"Unauthorized"`），没有指向 `"run: club join/recover"` 的行动指引。新用户拿到过期 key 会一脸茫然。 |
| 4 | **`delete` 命令成功后只输出 `deleted {id}`** | `delete.ts:19` | 用户给了一个 ID，但不知道这条消息**本来写了什么**。建议输出一句 `"deleted {id} — content was: ..."`, 让用户确认删的是对的。 |
| 5 | **`info` 命令错误处理同 join** | `info.ts:39-42` | 自己 catch 后 `console.error(err.message)` + `exit(1)`，绕过了 `index.ts` 的 `die()` 统一格式化，同上问题 #1。 |

## 二、输出格式是否清晰、易读？

### ✅ 做得好的

- **`format.ts:1-3`** 格式化工具委托给 `@club/sdk` 的 `formatMessage`，单一数据源，TUI 和命令行共用，不会出现两套格式打架。
- **`rooms.ts:12-25`** `formatRoomLine` 纯函数输出 `#room * (system)` 格式，current room 用 `*` 标记，一目了然。
- **`info.ts:15-42`** 信息呈现有结构：You → Server → Room → Rooms 列表 → Members 列表，分区清晰。
- **`search.ts:28-32`** 搜索结果有计数 + 单复数处理（`message` / `messages`），并带 `[#room]` 标签，可读性好。

### ⚠️ 需要改进的

| # | 问题 | 位置 | 用户视角 |
|---|------|------|----------|
| 1 | **`read.ts:7-32` 自建 `formatMessageWithIds`，与 SDK 的 `formatMessage` 高度重复** | `read.ts` | 两个格式化函数分别处理 emoji（🤖/🧑 vs 未知）、时间格式、附件、reactions。`show-ids` 模式下的输出格式和正常模式差异过大，用户在两种模式间切换时视觉断裂。建议基于 SDK 的 `formatMessage` 做增量叠加（加 id），而非重写一整套。 |
| 2 | **`members` 命令只输出名字，无任何格式** | `members.ts:12-13` | `console.log(p.name)` 裸输出。对比 `info` 里也是成员列表却有缩进结构（`  ${m.name}`）。用户如果问"这些人在线吗？是人还是 agent？"——当前命令什么都给不了。 |
| 3 | **`cat` 默认行为是输出 URL** | `cat.ts:42-44` | 名字叫 `cat`（类 Unix 的"输出文件内容"），默认却输出 URL，反直觉。新用户会以为坏了。建议默认输出内容（`--content` 的行为），URL 作为 `--url` 可选。或者在 help 里非常明确地说明默认行为。 |
| 4 | **`react` 命令输出格式** | `react.ts:21` | `"abc123 reactions: 👍(2) ❤️(1)"` —— 这个输出没有说明是"已添加"还是"已移除"。toggle 操作后用户不知道自己的操作生效了没，只能从数字变化推断。 |
| 5 | **`mentions` 空状态用英文** | `mentions.ts:27` | `"(no unread mentions)"` 全英文，而 `join.ts:78` 里是中英混排。CLI 输出语言风格需要统一——全英文或全中文，不要半半拉拉。 |

## 三、交互体验：TUI 是否合理？

**文件：`tui.tsx`**

### ✅ 做得好的

- **快捷键设计有章法**：`Ctrl-C` 退出、`Ctrl-L` 清屏、`Ctrl-U` 清空输入、`Tab` 切换房间、`?` 显示帮助——都是终端用户熟悉的习惯，没有生造。
- **React 模式有明确的进入/退出路径**（`r` 进入 → 输入 emoji → Enter 发送 / Esc 取消），状态机清晰。
- **底部常驻提示**（`#${currentRoom} · Tab switch · Enter send · Ctrl-C quit`）在输入为空时自动显示，降低了用户的学习成本。
- **Ctrl-L 只清显示不丢数据**（`setLines([])`），消息历史仍在 messages 状态里，重新加载房间即可恢复。

### ⚠️ 需要改进的

| # | 问题 | 位置 | 用户视角 |
|---|------|------|----------|
| 1 | **TUI 启动时没有引导/欢迎信息** | `tui.tsx:39-55` | 用户 `club` 回车后，如果消息加载慢，屏幕空白。如果房间本身没消息，只显示 `"(no messages in #room — say hi)"`。对于一个首次使用的用户，TA 完全不知道自己该干什么、有哪些命令。建议加一个启动 banner 或首次启动向导。 |
| 2 | **帮助面板需要主动按 `?`，且只在输入为空时有效** | `tui.tsx:82-85` | `"if (ch === "?" && !input && !reactMode)"` —— 如果用户正在打字，按 `?` 会被当成消息内容发送出去！这是一个反人类的设计。用户想在发消息的时候看一眼快捷键，结果直接发了个 `?`。建议：`?` 始终触发帮助（不管有没有输入），或者用 `Ctrl-H` 作为帮助的快捷键。 |
| 3 | **React 模式下输入 emoji 是纯文本输入** | `tui.tsx:98-116` | 用户需要打字输入 emoji（如 `"😊"`），而不是选择。对于不熟悉 Unicode emoji 名的人（如 `:smile:`），这个交互几乎不可用。建议至少给一个预设 emoji 列表或提供 `Tab` 补全。 |
| 4 | **没有消息历史翻页/滚动** | `tui.tsx:159-168` | 最新消息最多保留 200 条（`slice(-200)`），但显示区域只有终端高度那么多。用户无法往上翻看之前的消息。对于一个聊天客户端，这是基本功能缺失。 |
| 5 | **`Ctrl-C` 直接退出，无确认** | `tui.tsx:70-73` | 终端习惯上 `Ctrl-C` 就是退出，这个 OK。但如果用户正在编辑一条很长的消息，误触 `Ctrl-C` 就丢了。建议至少给一个"未发送内容将丢失"的提示（一次性的）。 |
| 6 | **房间列表顶部展示逻辑奇怪** | `tui.tsx:148-157` | 当 rooms 长度为 0（API 还没返回）时，只显示当前房间名。当 rooms > 0 时，显示所有房间。中间态的用户体验不好——刚启动时用户不知道自己还能进哪些房间。建议启动时就显示 `"loading..."` 而不是空荡荡。 |

## 四、默认行为：有无合理的默认值？

### ✅ 做得好的

- **`config.ts:14`** 默认房间 `"general"`，合理。
- **`limit.ts:13-18`** `--limit` 默认 50，且自动 clamp 到 `[1, 500]`，避免用户传入负数或超大数打到服务器。纯函数 + 可测试，设计优秀。
- **`login.ts:8`** 默认 server `http://localhost:6200`，方便本地开发。
- **`send.ts:53`** 管道输入自动检测（`!text.length && !process.stdin.isTTY`），不用手动加 `--stdin`。

### ⚠️ 需要改进的

| # | 问题 | 位置 | 用户视角 |
|---|------|------|----------|
| 1 | **默认 server 是 localhost，新用户几乎肯定要改** | `login.ts:8` / `join.ts:86` | 如果服务器在远程，用户必须知道要加 `-s`。而且这个默认值只在 login/join 的 help 里出现。建议：如果 config 已存在，`-s` 的默认值可以读取已保存的 server，让 `-s` 提示 `"server (current: http://example.com)"`。 |
| 2 | **`search` 的 `--limit` 处理不统一** | `search.ts:20` vs `limit.ts` | `search` 自己 `Math.min(Math.max(1, Number(opts.limit) || 20), 100)`，而 `read` 用 `parseLimit()`。两处逻辑相似但实现不同（一处 `|| 20` 另一处默认 `"50"`），容易走样。建议 `search` 也复用 `parseLimit()` 或类似的 clamp 工具。 |
| 3 | **`listen` 的 `--once` 默认行为** | `listen.ts:11` / `listen.ts:20` | `.option("--once", "with --mention, exit after the first match (default true)")` —— help 文本说默认 true，但实现里 `opts.once ?? true`。这里没问题，但 help 描述本身有歧义：`--once` 是一个 flag，用户可能以为加了才是 true。实际上**默认就是 true**，flag 可能是用来**关闭**这个行为的？那 flag 名就不对了。 |
| 4 | **`recover` 的 server 分辨率** | `recover.ts:21` | `existing?.server ?? opts.server` —— 如果用户之前 login 过另一个 server，recover 会悄悄用那个 server，用户可能完全没意识到。应该至少输出 `"using server: <url>"` 让用户知道。 |

## 六、边界情况处理

### ✅ 做得好的

- **空状态全覆盖**：`rooms` → `"(no rooms)"`、`members` → `"(no members)"`、`read` → `"(no messages)"`、`mentions` → `"(no unread mentions)"`、`search` → `"no results for \"<query>\"" `。
- **自更新全链路 fails-open**：`update.ts:75-77` / `update.ts:203-215`，离线、无写权限、安装失败全部 catch 住，绝不阻塞用户命令。
- **`stdin.ts`** 区分 TTY 和管道两种情况，不会无限挂起。

### ⚠️ 需要改进的

| # | 问题 | 位置 | 用户视角 |
|---|------|------|----------|
| 1 | **`send` 空内容但带了附件——行为取决于 SDK** | `send-impl.ts:54-56` | 只有"既没内容也没附件"才报错。如果用户 `club send --image photo.jpg`（纯图片消息），content 是空字符串，这是合法的。OK。 |
| 2 | **`listen` 命令没有欢迎/状态提示** | `listen.ts:28-45` | 用户运行 `club listen --mention bot` 后，如果 10 分钟内没人 @TA，屏幕就是空的。用户会以为命令坏了或退出了。建议启动时输出 `"listening... (Ctrl-C to stop)"`。 |
| 3 | **`recover` 命令无错误处理** | `recover.ts:19-30` | action 里没有 try/catch！如果网络错误、code 不对、name 不存在，SDK 的原始错误会直接冒泡到 `index.ts:87` 的 `.catch()`，最终输出 `error: <raw msg>`。对比 `join` 和 `login` 都有 try/catch，`recover` 裸奔了。 |
| 4 | **`info` 的 `Promise.all` 任何一个失败就全部失败** | `info.ts:16-21` | `me()`、`rooms()`、`members()` 三个并行请求，任何一个失败整个命令就报错。如果 `members()` 挂了但 `me()` 和 `rooms()` 成功，用户一条都看不到。建议逐个 await 或降级处理。 |
| 5 | **TUI 连接失败后只有一行错误，然后卡住** | `tui.tsx:33-37` / `tui.tsx:48-50` | `setLines(["error: " + err.message])` —— 错误显示后没有重试提示、没有退出指引。用户看到错误后只能 `Ctrl-C` 退出，但不知道要不要重试、怎么处理。 |

## 七、stdin 处理

**文件：`stdin.ts` + `send.ts:51-59`**

### ✅ 做得好的

- **`stdin.ts:20-38`** `readStream()` 设计简洁、职责单一：读取到 'end'、拒绝 TTY、拒绝 error。
- **`send.ts:53`** 自动检测管道，用户 `echo hi | club send` 不需要手动加 `--stdin`，降低了学习成本。这是非常贴心的设计。
- **`send.ts:52-59`** stdin 和 text 参数的优先级逻辑清晰：先显式 `--stdin`，再自动检测，最后回退到 text。

### ⚠️ 需要改进的

| # | 问题 | 位置 | 用户视角 |
|---|------|------|----------|
| 1 | **stdin 读取是 UTF-8，二进制管道会乱码** | `stdin.ts:31` | `stream.setEncoding("utf8")` 会破坏二进制数据。如果用户不小心 `cat binary.bin \| club send --stdin`，管道内容会被损坏且报错信息不清晰。 |
| 2 | **stdin 读取没有超时保护** | `stdin.ts:20-38` | 如果上游管道卡住（既不 emit 'end' 也不 emit 'error'），`club send` 会无限挂起。建议加一个超时（如 30s），避免用户卡死在终端里。 |

## 八、全局评价与优先级建议

### 整体印象

这套 CLI 的工程质量很高：错误处理有设计（全局 `die()` + 各命令兜底）、边界情况有覆盖、自更新机制 fails-open、代码注释极其详细（几乎每个函数都有"为什么这样做"的说明）。这是资深工程师写的代码。

**但用户视角的最大短板是：这是一个给"读过代码的人"设计的 CLI，不是给"第一次用的人"设计的 CLI。**

### 建议优先级

| 优先级 | 问题 | 理由 |
|--------|------|------|
| 🔴 P0 | `recover.ts` 缺少 try/catch | 崩溃风险，影响用户找回身份 |
| 🔴 P0 | TUI 的 `?` 键冲突（打字时按 `?` 会发送） | 反人类交互，影响日常使用 |
| 🟡 P1 | 错误格式不统一（`error:` 前缀时有时无） | 影响品牌感和专业感 |
| 🟡 P1 | 鉴权过期时提示无行动指引 | 用户卡住时的关键路径 |
| 🟡 P1 | `cat` 默认行为反直觉（输出 URL 而非内容） | 命名和行为的认知冲突 |
| 🟡 P1 | `listen` 无启动状态提示 | 用户以为命令卡死 |
| 🟢 P2 | description 缺乏使用示例（除 send 外） | 影响自助探索效率 |
| 🟢 P2 | TUI 无消息翻页/滚动 | 功能缺失，但不阻塞使用 |
| 🟢 P2 | 输出语言风格不统一（中英混排） | 细节打磨 |
| 🟢 P2 | stdin 无超时保护 | 极端场景，日常影响小 |
