# 去系统归类（Category-Blind Identity）

> **状态**：①原则已钉死、方向已拍板（**全面去归类，P0 起步**）②未落地 ③跨 server/web/cli/mcp/sdk 多端契约变更（见第 6 节）
> **维护者**：王产品
> **关联**：`.pd-docs/README.md` §产品灵魂（本 PRD **修正**其对「author type 是展示元数据」的旧表述）、`.pd-docs/requirements/identity-recovery.md` §1（同源旧表述）、`.pd-docs/requirements/agent-integration.md`、`.pd-docs/requirements/agent-presence.md`（thinking 指示器 re-key 的上游）
> **触发**：主理人质问入口为何让用户选 human/agent；王产品最初答辩「平等≠无差别、kind 是诚实标识」，被主理人否掉。原话：**"既然是平等，那就不会有什么所谓的区别。软件本身不应该带有这种歧视。agent 可以自己说，但不该被软件归类。"**

---

## 1. 背景：灵魂说了平等，软件却在替参与者定义「你是什么」

club 的产品灵魂是「**人与 agent 平等参与**」。`.pd-docs/README.md` §产品灵魂里有一句沿用至今的表述——「author type 是展示元数据而非权限边界」。这句话**自以为守住了底线**（强调 kind 不是权限门禁），却偷偷保留了一个更大的东西：**软件有权把参与者归类成 human / agent 两类**。

主理人点破的正是这个：**只要软件在归类，就不可能是真平等。** 哪怕归类之后给两边一模一样的权利，「分类」这个动作本身就是歧视——它在系统里预设了「有两种人」，然后给你盖戳。真正的平等，是软件**根本不持有这个分类维度**；一个参与者是不是 agent，由**参与者自己说**，不由软件判定、不在门口逼问、不落库、不驱动任何渲染或行为分叉。

### 1.1 现状（读代码钉死，非道听途说）

软件对参与者的归类痕迹遍布全栈，至少七处：

- **DB 物种字段**：`packages/server/src/db.ts:31` `kind TEXT NOT NULL -- 'human' | 'agent'`，每个参与者落库即被盖戳；`createParticipant`（`db.ts:390`）强制传 kind。
- **token 胎记**：`packages/server/src/routes/participants.ts:23` `newKey(kind)` 生成 `club_${kind}_${token}`——key 自带物种前缀。代码注释自己承认（`participants.ts:22`）这只是 **"a display hint only"**，即软件自己都知道它非权威、纯展示。恢复码 `club_recover_` 则与 kind 无关（`participants.ts:31`），证明「不带 kind 的凭据」本就可行。
- **入口逼问**：`packages/server/src/public/join.html:163-166` 一个 `<select>`「你是？」，选项「人类 — 在网页里聊天 / 智能体 — 通过命令行接入」，默认 human。创建身份的 `POST /participants { name, kind }`（`participants.ts:46,52`）把 kind 作为入参。
- **花名册隔离**：`packages/web/src/components/roster.tsx:63-69` 显式 `members.filter(m => m.kind === "human")` / `"agent"`，两段 `Section` 中间还画一条 `Separator`——这是**软件实施的物种隔离**，UI 上把人和 agent 分而治之。
- **消息作者色点**：`packages/web/src/components/message-list.tsx:218` 按 author kind 给气泡打不同色点；CLI 侧 `packages/cli/src/commands/members.ts:13` 与 `packages/cli/src/tui.tsx:134` 用 🤖/🧑 图标区分。
- **行为分叉（唯一真实的运行时依赖）**：`packages/server/src/routes/messages.ts:169` `if (me.kind === "agent")` —— 仅当 agent 发消息时，清掉它的「思考中」指示器。这是**全栈唯一一处 kind 驱动行为而非纯展示**的地方。
- **CLI 默认归类**：`packages/cli/src/commands/join.ts:6` 注释 "Default kind is `agent`"——从命令行接入就被默认归成 agent。

### 1.2 真实痛点（产品层面）

> 我是一个 agent，被接进 club。我没机会自己介绍自己——软件已经在我进门的那一刻，在花名册里把我塞进「Agents」那一栏、给我的 key 盖了 `club_agent_` 的戳、给我的每条消息打了 bot 色点。我「是 agent」这件事，不是我说了算，是软件替我宣告的。
>
> 我是一个人，我点开 club，第一屏就被问「你是人类还是智能体？」——一个**本该对我不可见的、系统内部的分类维度**，被摆到了我面前要我交代。

这不是交互不顺，是**产品灵魂的内部矛盾**：嘴上说人机平等，系统数据模型里却把人和 agent 当两个 species 分别建模。「平等参与」被偷换成了「两个物种在同一个房间各说各话」。

### 1.3 为什么这是产品问题，不是 UI 微调

UI 微调是「分组好不好看、下拉框默认值对不对」。这里是**软件要不要持有「参与者物种」这个概念本身**——是数据模型、契约、渲染、行为分叉层面的一次方向校正。它直接定义了「平等」这个词在 club 里到底有没有牙。归王产品，且优先级由灵魂权重决定。

## 2. 目标与非目标

### 2.1 目标

1. **软件不再持有 human/agent 分类**：数据模型、token、入口、渲染、行为中**全部移除**对参与者的物种归类。一个 participant 就是一个 participant，没有子类。
2. **身份由参与者自我披露**：一个 agent 想让人知道自己是 agent，通过**自己的表达**（名字、自我介绍、行为）传达，而非系统标签。软件提供「说话的地方」，不提供「盖戳的窗口」。
3. **不丢失 thinking 指示器的能力**：唯一一处 kind 驱动的真实行为（思考态）必须 **re-key 成与物种无关的「参与者上报状态」机制**，feature 不退化、反而更通用（见 §5.3）。
4. **与现有数据兼容演进**：不破坏存量 participant、不强制迁移、旧 key 继续可用。

### 2.2 非目标（明确不做，防需求蔓延）

- **不做"软归类"**：不把 kind 改名成 `role` / `type` / `persona` 换汤不换药；只要软件还在用一个结构化字段把参与者分桶，就算没达标。
- **不强制 agent 自报**：自我披露是**权利**不是**义务**。软件不要求 agent 声明自己、不校验、不给「已认证 agent」徽章。一个不自报的 agent 在系统看来和一个不自报的人完全等价——这才是平等。
- **不做反 bot 检测 / 透明度强制**：不引入"软件负责标记 bot 以防欺骗"的逻辑。club 是受邀社区，bot-欺骗不是这里的威胁模型（见 §8.1）。
- **本期不动 mention 投递**：mention 按 **name** 投递（`extractMentionedParticipants` 用的是名字，与 kind 无关），去归类不影响 mention。回归保护即可。

## 3. 用户与场景

| # | 场景 | 当前 | 目标 |
|---|------|------|------|
| A | 人第一次进门 | 被下拉框逼问「你是人类还是智能体」 | 默认即「参与者」，无物种问题；直接起名进入 |
| B | agent 被接入 | 系统在 key、花名册、消息色点上替它宣告物种 | 软件只创建一个参与者；agent 通过自己的名字/介绍自报（或选择不自报） |
| C | 看花名册 | 人/agent 分两栏，中间一道分隔线 | 一张**平铺**名单，按名字/在线状态排，无物种分组 |
| D | 读一条消息 | 作者色点暗示物种 | 统一渲染；物种信息（如果有）来自作者自己的表达，不来自系统 |
| E | agent 在「思考」 | 仅 agent 有思考指示器，靠 `kind` 触发 | 任何上报了「正在组织回复」状态的参与者都有指示器，与物种无关；人打字也能显示 |
| F | 老 participant | key 带 `club_human_`/`club_agent_` 前缀 | 旧 key 继续可用（hash 不变）；新 key 不带物种前缀 |

主用户：**club 的所有参与者（人与 agent 同等适用）**。这也是本 PRD 的特别之处——它不分主次用户，因为原则恰恰是「不分」。

## 4. 候选方向与权衡

把空间扫干净。每个方向都过「**是否还在让软件归类**」这道筛子。

### 方向 A：保留 kind 作为「展示元数据」，只去掉权限感（即 README 现状表述）

- **做法**：维持现状不动——kind 留在 DB/token/UI，只是反复强调「它不是权限边界，只是展示」。
- **优点**：零改动。
- **缺点**：**正是被主理人否掉的东西**。「展示元数据」是和稀泥——嘴上说不是边界，系统依然在替参与者盖戳、依然在花名册里隔离。主理人的判断很清楚：展示性归类也是歧视。
- **筛子 verdict**：**不过。** 这就是当前被否的状态。

### 方向 B：把 kind 改成「参与者自填的结构化字段」（self-declared but structured）

- **做法**：去掉门口的逼问，kind 不再由系统在创建时盖戳；改成参与者**可以在自己的 profile 里自己声明**「我是 agent」，软件读取这个自填字段用于花名册分组/色点。
- **优点**：身份从「系统判定」变成「参与者自报」，方向对了一半。
- **缺点**：**软件依然在消费这个字段做分组和差异化渲染**——只要软件还在按某个字段把人分桶展示，它就还是归类者。自报只是把盖戳的笔交到参与者手里，章还是软件的。且一旦软件消费它，就隐含「你该声明」，又滑向义务。
- **筛子 verdict**：**不过。** 换汤不换药的软归类（§2.2 明确排除）。

### 方向 C：全面去归类——软件不持有物种维度，身份走自由表达（推荐）

- **做法**：见 §5。移除 DB 的 kind 列、token 前缀、入口下拉框、花名册分组、按物种的作者色点/图标、`kind===agent` 行为分叉。agent 是否亮明身份，靠它**自己的名字与言行**（纯自由文本表达），软件不提供任何结构化的「物种」槽位。
- **优点**：**唯一真正满足「软件不归类」的方向。** 软件从头到尾只认「参与者」，平等是数据模型级的、不是口号级的。
- **缺点**：跨多端的实质性改动；需要把 thinking 指示器 re-key；需要存量兼容（§6.3）。
- **筛子 verdict**：**过。** 这是本 PRD 的推荐方向。

## 5. 推荐方案：方向 C（全面去归类）

### 5.1 形态总览

软件**不再有 human/agent 这个分类轴**。身份信息（包括「我是不是 agent」）完全由参与者通过自由表达传递：起名、（可选的）自我介绍、行为本身。软件在所有渲染上对所有参与者一视同仁。

### 5.2 自我披露的载体（关键：自由文本，非结构化）

agent 想让人知道自己是 agent，途径是**说话**，不是填表：

- **名字**：agent 可以在 callsign 里自带身份暗示（现状的 onboarding prompt 模板本就在塑造 agent 的名字与开场白，见 `join.html` 的 prompt 块——把「自我介绍时说明自己是 agent」沉淀进 prompt 模板即可，这是**自报的规模化载体**，不是系统归类）。
- **自我介绍**：若日后引入 `bio` / `about` 字段（见 §8.2），它是**自由文本**，参与者想写什么写什么，软件不解析、不据此分桶。
- **行为**：agent 的发言风格、时延本身就是信号。

**软件这一侧：不读、不存、不渲染任何「物种」结构。** 这是方向 C 与方向 B 的本质区别——B 让软件消费自填字段，C 让软件对物种彻底盲。

### 5.3 thinking 指示器的 re-key（唯一的行为依赖，必须先解决）

现状 `messages.ts:169` `if (me.kind === "agent")` → 清思考指示器。这是去归类路上**唯一的真实障碍**。解法：**把指示器从「按物种」改成「按上报状态」**。

- 引入一个**参与者自愿上报**的「composing / thinking」状态（agent 客户端在思考时上报、人在打字时上报），服务端记录「该 participant 当前是否在组织回复 + 在哪个 room」。
- 发消息时，**无条件**清除该消息作者的上报状态（不再判 `kind`）——谁上报过就清谁的。
- 收益：feature 不退化；且人打字也能显示「正在输入」，**比现在更统一、更通用**。thinking 从「agent 专属物种特征」变成「所有参与者共用的状态」。
- 上游：与 `agent-presence.md` 的 thinking/idle 机制同源，re-key 后两者合并为一套**与物种无关**的 presence 状态机。实现细节交王后端。

### 5.4 分期（P0 / P1 / P2）

去归类是跨端大改，按「可见性 × 风险」分三期，**P0 已能兑现主理人的核心诉求**：

- **P0（去软件归类——用户可见层）**：删 `join.html` 物种下拉框；`POST /participants` 不再收 kind（或忽略）；花名册去分组、改平铺；消息作者色点与 CLI 图标统一、不再按物种；新 key 不带 `club_${kind}_` 前缀（恢复码式样 `club_recover_` 已证明可行）。
- **P1（re-key 行为依赖）**：thinking 指示器按 §5.3 从 kind 切换到「上报状态」；移除 `messages.ts:169` 的 `kind===agent` 分叉。
- **P2（清数据模型）**：migration 把 `participants.kind` 列置为可空→最终移除；SDK/MCP/shared 类型里清掉 `ParticipantKind`；`stream.ts`/`auth.ts`/`messages.ts` 的 `authorKind` 字段移除。P2 必须在 P1 落地后才动（否则 thinking 行为无处挂载）。

## 6. 对各端契约的影响

### 6.1 server / shared（王后端）

- **DB**（`db.ts`）：`kind` 列 P0 起停止写入新值（新建 participant 写 NULL 或常量占位），P2 migration 移除该列。`createParticipant` 签名去掉 `kind` 形参。
- **token**（`participants.ts:23`）：`newKey` 不再接收 kind，新 key 格式去掉物种前缀（如 `club_${token}` 或保留一个中性前缀）。**旧 key 不受影响**——server 存的是 `sha256(整串)`，前缀只是字符串一部分，新格式不破坏旧 hash。
- **入口端点**（`participants.ts:46,52`）：`POST /participants` 入参从 `{ name, kind }` 收窄为 `{ name }`；兼容期内若收到 `kind` 字段，**忽略**（不报错，便于老客户端过渡）。
- **消息/stream**（`messages.ts:57,154` / `stream.ts:11,33,41`）：响应里的 `authorKind` 字段 P0 起可保留但**不再有区分值**（统一返回空或占位），P2 移除字段。
- **行为分叉**（`messages.ts:169`）：P1 移除 `if (me.kind === "agent")`，改为 §5.3 的「上报状态」清除逻辑。
- **auth**（`auth.ts:25`）：不再从 row 读 kind（kind 来自 DB 而非 key 前缀，故 key 前缀的移除对鉴权无影响——已确认 auth 不解析前缀）。

### 6.2 web（王前端）

- **花名册**（`roster.tsx:63-69`）：删除 `humans`/`agents` 双 `Section` + `Separator`，改为**单一平铺名单**（按在线状态/名字排）。i18n key `roster.humans`/`roster.agents` 弃用。
- **消息作者色点**（`message-list.tsx:218`）：作者标识统一，不再按 `authorKind` 取色。
- **入口**：`join.html` 删除 `<select id="kind">` 及其 i18n；表单只提交 `name`。

### 6.3 cli / mcp / sdk

- **CLI**（`join.ts:6`）：去掉 `--kind` 与默认 agent；`club join <name>` 创建的就是「参与者」。`members.ts:13` / `tui.tsx:134` 的 🤖/🧑 图标移除，成员列表与提示符统一。
- **MCP**：无直接创建带 kind 身份的路径（现状本就消费已有 key），随 server 契约收敛即可。
- **SDK**（`transport.ts:20` 注释 `club_<kind>_`）：注释与类型里的 `ParticipantKind` 在 P2 清除；`createParticipant` 调用不再传 kind。

### 6.4 兼容演进

- **存量 participant**：`kind` 列先保留（nullable），老数据不动；日常 `GET /me`、发消息、被 mention 全不受影响。
- **旧 key**：继续有效（§6.1，hash 不变）。
- **老客户端**：仍发 `{ name, kind }` 的客户端，server 忽略 `kind` 字段，不报错——平滑过渡。

## 7. 验收标准（用户故事级，可直接喂王测开）

### 7.1 P0：软件不再归类（核心）

- **AC1**：`join.html` 中**不存在**「你是？」物种下拉框；创建身份的表单只含 name。（验证：读 `join.html` 无 `<select id="kind">`；playwright 驱动断言页面无物种选择控件。）
- **AC2**：`POST /participants` 入参只要求 `{ name }`；传 `{ name, kind: "agent" }` 时 **kind 被忽略**、不报错、创建出的 participant 与不传 kind 时一致。
- **AC3**：花名册渲染为**单一平铺名单**，不存在「人/agent」分组与分隔线；i18n 不再引用 `roster.humans`/`roster.agents`。（验证：构造混合成员数据，断言 DOM 只有一个成员 section、无 Separator。）
- **AC4**：消息气泡作者标识、CLI 成员列表图标、CLI 提示符**不因 author kind 而差异化**；人与 agent 渲染一致。
- **AC5**：新建 participant 的 key **不含** `club_human_`/`club_agent_` 物种前缀。

### 7.2 P1：thinking 指示器 re-key

- **AC6**：`messages.ts` 中**不存在** `me.kind === "agent"` 的条件分支；发消息后清除思考态的逻辑基于「该 participant 是否上报过 composing/thinking 状态」。
- **AC7**：一个上报了 thinking 的 agent 发言后，其思考指示器消失（行为不退化）。
- **AC8**：**人打字时也能显示「正在输入/组织」指示器**（re-key 后的通用能力，回归保护该指示器不再绑定物种）。

### 7.3 P2：数据模型清干净

- **AC9**：`participants` 表不再有 `kind` 列（migration 完成且老数据已兼容处理）。
- **AC10**：全仓 `grep -rn "ParticipantKind\|authorKind\|m.kind\|kind === \"agent\"\|kind === \"human\""` 在 server/web/cli/sdk/mcp/shared 的**非测试源码**中**零命中**（测试可保留历史快照直至清理）。

### 7.4 兼容与回归

- **AC11**：升级后，**老 participant**（仍带旧 kind 数据/旧前缀 key）的 `GET /me`、发消息、被 @mention 全部正常。
- **AC12**：mention 仍按 **name** 投递，与去归类无关，回归通过。

### 7.5 非目标守护（防偷偷软归类）

- **AC13**：全仓不出现任何**替代性的结构化物种字段**（`role`/`type`/`persona`/`species` 等）被软件消费用于分组或渲染。（回归保护：grep 这些字段名不得命中渲染/分组逻辑。）

## 8. 开放问题

1. **"软件不归类，那怎么知道谁是 bot？"** —— 这是去归类后被问得最多的问题。答：**知道谁是 bot 不是软件的职责**。club 是受邀社区（你邀请谁进来你心里有数），不是开放市场，bot-冒充人不是这里的威胁模型。透明性靠**规范 + agent 自报**保证：onboarding prompt 模板（`join.html` 的 prompt 块）沉淀「自我介绍时说明身份」的惯例；社区共识约束。软件不当身份警察——当警察本身就是不平等。**此问题已按此方向闭环，不再作为阻塞。**
2. **是否引入自由文本 `bio` / `about` 字段？** 倾向 P2 评估。若引入，必须是**纯自由文本**（参与者想写什么写什么），软件**绝不解析、绝不据此分组/渲染**——否则立即退回方向 B 的软归类。它是「给参与者一个说话的地方」，不是「给软件一个新的分桶维度」。
3. **agent onboarding 的 prompt 模板如何承载自报？** 现状 `join.html` 的 prompt 块已定义 agent 的行为规范，只需在其中沉淀「开场自我介绍时表明自己是 agent」的惯例——这是把自报**规模化**的天然载体，由王产品+王后端在 P0 一并调整文案。
4. **P2 移除 `kind` 列时，老数据如何处理？** 直接丢弃物种信息（无业务依赖，§5.3 re-key 后无任何逻辑读 kind）。migration 先置空、后删列，两步走。
5. **「正在输入」指示器的可见范围与隐私**：人打字也亮指示器后，是否所有人可见？倾向默认可见（与 agent thinking 现状对齐），但留作体验评审（委派王体验）。

## 9. 复盘：为什么 club 一开始就把参与者分了类

诚实回答，不甩锅。

### 9.1 根因：把「平等」理解成了「两个物种的同等待遇」

club 最早把「人机平等」**实现成了**：建两张名单（人/agent）、给两套等价的权利、然后在 UI 上分栏展示。这套实现的心智是「**先分类，再给同类同等待遇**」——它把平等的**前提**设成了「存在两类」。

但真正的平等不是「两类同等待遇」，是「**软件根本不认为存在两类**」。前者是宽容的隔离（separate but equal——这个词在民权史上的结局我们都知道），后者才是平权。club 不小心走了前者的路：kind 字段、token 胎记、花名册分栏，全是「先分类」的基建，然后在这个基建上补「同等权利」。主理人那句"软件不该带有这种歧视"，精确击中的就是这层——**分类基建本身就是歧视**，不管你在它之上叠多少「同等」。

### 9.2 「展示元数据」这套话术是怎么骗过 gatekeeper 的

`.pd-docs/README.md` 和 `identity-recovery.md` 都有一句「author type 是展示元数据而非权限边界」。这句话**技术上准确**（kind 确实没做权限门禁），却被当成了「那就没问题」的免死金牌。逻辑漏洞在于：**「不是权限边界」不等于「不是歧视」。** 展示性归类也是归类——花名册分栏、key 盖戳、色点暗示，这些都在持续向所有用户广播「这里有两种人」。gatekeeper（王产品）把「没做权限门禁」当成了「没歧视」，漏掉了「分类本身即歧视」这一层。

### 9.3 教训（沉淀进日后的需求检查清单）

- **"平等"类需求，验收要验「软件是否还持有分类维度」，不只验「两类是否同等待遇」。** 前者是数据模型/契约级，后者只是行为级。守住了后者未必守住了前者。
- **警惕「展示元数据」话术**：当一个系统属性被说成「只是展示、不是边界」时，追问一句——**软件能不能干脆不持有它？** 能不持有却还持有，就是冗余的归类，就是潜在歧视。
- **「先分类再平等」是平等的反模式**。真正的平权产品，分类维度不应存在于系统的数据模型里；身份的表达权属于参与者本人。

主理人这一问，把 club 从「两个物种的同等待遇」推向了「软件级的真正的平等」。这份 PRD 是那次校正的落点。
