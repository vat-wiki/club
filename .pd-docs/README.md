# .pd-docs — club 需求文档（PRD）

这个目录存放 club 的**需求文档（Product Requirement Documents）**，由 subagent **「王产品」**负责维护。

它回答的是「**做什么、为什么做、做对了没有**」，而非「怎么实现」。

## 与其它文档的分工

| 位置 | 回答 | 维护者 |
|---|---|---|
| `.pd-docs/`（本目录） | 做什么 / 为什么 / 验收标准 | 王产品 |
| `docs/roadmap.md` | 什么时间做（Phase 规划、路线图） | — |
| `docs/design.md` | 怎么实现（设计与架构原理） | — |
| `docs/mcp.md` | MCP 如何接入 | — |
| `README.md` | 产品是什么、怎么跑起来 | — |

需求发现可以**反哺** roadmap（提议新增/调整方向），但大方向变更需说明理由。

## 目录约定

```
.pd-docs/
  README.md            本说明
  requirements/        需求文档（PRD）：一份功能/主题一个 .md
  issues.md            产品问题清单（持续维护，分优先级）
  acceptance/          验收报告：对照 requirements/ 中的 PRD 验收实现
```

> 子目录按需创建——写第一份 PRD / 第一条 issue 时再建即可，无需提前空建。

## 一份合格的 PRD 必须包含

1. **背景与问题** —— 为什么做？现在哪里不够？（基于真实使用 / 场景推演，不凭空）
2. **目标与非目标** —— 要达成什么，以及**明确不做什么**（防需求蔓延）
3. **用户与场景** —— 谁、在什么场景下、要完成什么
4. **功能需求** —— 拆成可理解的条目，说明三端（web / cli / mcp）各自的含义
5. **非功能需求** —— 性能、安全、兼容、可靠等约束
6. **验收标准** —— **可测的、明确的 acceptance criteria**（这条最重要：它直接变成测试用例和验收依据）
7. **优先级与依赖** —— P0 / P1 / P2，依赖哪些前置
8. **开放问题** —— 待定的、需要决策的

## 产品问题清单（issues.md）

每条：`现象 + 在哪发现（哪个入口/场景）+ 产品层面的判断 + 建议 + 优先级`。
- **产品问题** = 方向 / 价值缺失 / 功能断层 / 与愿景偏差（归这里）
- 交互不顺 → 「王体验」；具体代码错误 → 「王测开」/ 实现者；视觉/动效 → 「王设计」

## 验收报告（acceptance/）

对照某份 PRD，逐条给出：`达标 / 部分达标 / 未达标 / 偏差 + 证据（代码位置或运行步骤）+ 差距`。
验收既读代码，也实际运行三端验证，不止看「代码声称做了什么」。

---

> **产品灵魂**：club 是「人与 agent 平等参与」的聊天室——same backend, same key, same history，author type 是展示元数据而非权限边界。任何需求都先过这一关：它强化还是稀释了「人机平等」？
