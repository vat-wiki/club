## 2026-07-20 09:47 — 方向：性能（Performance）

- 为 `extractMentionedParticipants`（@-提及解析）新增模块级 name→participant Map
  惰性缓存（`mention.ts`）：按参与者数量 O(n_participants) 仅在首次 / 缓存
  失效时构建，消息写入 hot path 单次调用降为 O(content 长度 + @ 出现次数)。
  同步提供 `invalidateParticipantNameMap()`，由 `participants.ts` 在创建 / 凭证
  轮换 / 恢复 / 恢复码轮换共 4 处调用点与 `invalidateParticipantNamesCache()`
  同步失效，保证消息写入路径永不读到过期参与者名单。
- 将 `db.ts` 的 `getAllParticipantNames()` 改为返回冻结的稳定引用
  （`Object.freeze([...])`），让 mention 模块的引用恒等性检查（`roster !==
  _cache.roster`）在 hot path 命中 — 每轮消息发送复用同一个 Map，只有失效
  时才会重建。
- 附带完成前一轮遗留的 `messages.ts` 改动：`parseAttachments()` 加 512 项 LRU
  缓存，按原始 JSON 串键共享解析后的 `MessageAttachment[]` 引用；纯文本行（
  `raw === null/''`）保留无缓存的快速路径（绝大多数消息无附件）。
- 补测试：`mention.test.ts` 加 `beforeEach(invalidateParticipantNameMap)` 防止
  缓存跨用例泄漏；新增 2 个用例验证缓存重建（同尺寸不同参与者 / 显式失效恢复
  行为）。
- 验证：server `tsc --noEmit` 0 error；`vitest run` 242/243 通过（me.mentions
  batch mark-read 1 个失败，改动前已存在，非本次回归）；mention.test 18/18 通过。
